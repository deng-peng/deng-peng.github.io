chrome extension web request

整理一下chrome extension的几种js和请求网络的能力

Content Scripts

  • 需要权限 cross-origin permissions
  • 无法跨域 XMLHttpRequest cannot load http://xxx. Origin chrome-extension://xxx is not allowed by Access-Control-Allow-Origin.
  • 可以使用jsonp绕过cross domain
  • 如果所在parent页面是https的，那么http请求会被block. 只能使用background script来处理

Background Scripts

  • 需要权限 cross-origin permissions
  • 不存在跨域的问题
  • 一般业务数据都来自页面，所以需要通过content script传递数据。通信在content处调用chrome.extension.sendRequest

laravel执行migrate:rollback时提示Class 'CreateXXXTable' not found

在使用laravel时，对数据库的初始化操作，经常会使用migrate。
今天遇到一个奇怪的问题，在添加一个table后，字段不太合适，稍微调整想刷新一下，但是php artisan migrate总是出现如下提示

[Symfony\Component\Debug\Exception\FatalErrorException]
Class 'CreateXXXXXXXXXTable' not fount

但是这个Table其实就是我刚刚创建的，怎么会不存在呢。经过一番搜索，发现这可能是artisan的一个小问题，在添加的适合没有自动执行composer添加到lock里，所以就找不到映射了。
那么解决办法也很简单，先执行一下composer dump-autoload就好了，在新版本的composer里，可以省掉分割线，直接执行
composer dumpautoload即可。

记录一个artisan serve 总是异常退出的错误

使用laravel自带的web服务器artisan serve遇到的一个奇怪的问题，在访问一个路由时，总是自动退出，没有任何错误提示。

在controller的入口处做输出，发现执行不到这里。在router里记录，可以输出。

查看laravel的日志，也不会记录这个serve的错误信息。多次试验以后，发现只有这一个路由访问会出现这个问题，怀疑是此处的问题，在controller的构造函数处echo信息，依然不会输出。

逐层往下看，发现在底层的一个__construct了，依赖注入了上层的对象，形成了循环引用，移除这个注入，恢复正常。

解决mac下python无法连接mysql数据库的问题

最近在mac上修改自己的一个python爬虫，结果总是无法连接到本地的mysql数据库，但是本机的php却连接正常，很是奇怪。

网上搜索一通，基本都是类似修改配置的问题，基本毫无头绪。只好反思php为什么可以正常使用呢？

因为新近在mac上使用php+mysql，系统已经自带了apache和php，所以直接使用了Sites目录作为web目录。

但是PHP无法通过localhost来访问mysql，只能通过127.0.0.1，一番搜索，发现是mysql的sock问题。

mysql.sock文件是服务器与本地客户端进行通信的Unix套接字文件，其默认位置是/tmp/MySQL.sock, 拷贝至/var/mysql/mysql.sock目录，这时localhost已经可用，但是127.0.0.1却不可用了，想来应该是类似windows的host的对应机制，不过也没有去深究了。

到此，想来是不是也是mysql.sock的问题呢?

因为我的python项目使用了PyMySql，查找文档，修改了一下连接字符串

db = MySQLDatabase('xxx', user='root', password='1234', charset='utf8', unix_socket='/var/mysql/mysql.sock')

再次尝试，一切正常了。

使用fis3和Frozen UI构建前端项目

#FIS3 , 百度推出的前端工程构建工具。

##安装

npm install -g fis3

###好用的地方

  1. 资源定位，自动检查替换项目中的图片,js,css资源，通过match的方式替换为你想要的位置。
  2. js，css压缩，加md5，自动替换。
  3. 文件监听，浏览器自动刷新。

###遇到的小坑
有很多时候本地的项目都使用相对路径，但是fis的资源替换一定给你转成绝对路径，没有域名的情况下很不方便，需要在fis-conf里面各种配置。
后来发现原来fis团队提供了一个相对路径的插件fis3-hook-relative

####安装插件

npm install -g fis3-hook-relative

####使用插件

// 启用插件
fis.hook('relative');
// 让所有文件，都使用相对路径。
fis.match('**', {
  relative: true
})

#Frozen UI，腾讯推出的简单易用，为移动端定制的前端框架

##安装

npm install frozenui

###好用的地方

  1. 提供了各种移动端常用控件和样式，以及弹窗、提示条等插件。
  2. 几种布局，也很容易扩展。
  3. 圆角、占位、截断、留白等考虑的很细很好用。

viewport的那些事

虽然写过很多页面，但几乎所有header里meta都是这么写的

<meta name="viewport" content="width=device-width, initial-scale=1">

有时在移动页面里会加上禁止缩放

<meta name="viewport" content="width=device-width,initial-scale=1,maximum-scale=1,user-scalable=no" />

这也是html5 boilerplate的推荐写法，平时并没有深究过。

最近在编写一个html5的页面时，设计稿是按照640px的宽度制作，但并不要求是响应式布局，写完以后放在手机一看，发现各种地方的宽度都不太对，在header里设置

<meta name="viewport" content="width=640,maximum-scale=1,user-scalable=no" />

所有页面恢复正常，但这也让我决定去深究一下这里的原理。

其实页面显示不正常的原因，就是css里的px和设备的px没有对应上，也就是1px≠1px。这里怎么理解呢，可以认为一个margin:10px的地方，css的距离是10px，当你在设备上缩放页面时，虽然看到的距离（设备px）已经非常大了，但是css的尺寸依然是10px。

常见的设备device-width如下：

  • mdpi 320
  • hdpi 480
  • xhdpi 640

像我这样，强制设置为640时，在小的设备上，会自动缩小显示，保持布局正确。

但是推荐做法依然是设置width=device-width，有些朋友可能会误解这样设置了那不就是说明页面和屏幕尺寸一一对应了吗，为什么里面尺寸显示的不对呢，这里有一个像素密度的原因，也就是devicePixelRatio（可理解为移动App开发中的dip），这个值一般为1-2之间，iPhone5为2，iPhone6为3。
这个值可以表示一个设备px对应几个css的px。

viewport还有一个参数是target-densitydpi=device-dpi，如果这样设置，手机会按照真实的像素数目来渲染，这样也会实现1px=1px。

总结一下python爬虫常用的几个库

因为涉及到的几个抓取程序都是定时执行，或者一次性执行，并不太考虑持续执行时的性能问题，所以更偏重选择使用简单的库，over。

一种简易的mysql缓存方法

最近的一个项目，需要进行复杂的sql查询后给首页提供一组数据显示。而且首页可以滑动按天浏览，也就是每次切换都有比较大的sql查询。

在我本地环境运行是很快的，但是放在服务器上就非常慢了，切换日期一般需要等待5s以上，可能是因为我用了非常屌丝的最低配置linode服务器。

首页的浏览速度是一定要保证的，这里我本来打算增加一个缓存库，每次抓取完当天数据以后生成一个首页的缓存表；但是产品还处于迭代阶段，并不知道随后的结构会如何变化，经过考虑，采用一个简单的缓存机制：

使用php的serialize和unserialize来进行缓存

具体就是在查询数据后，采用一个key ：date value：sqlResult 的结构来存储到一个缓存表中，这个表和其他表都不存在关系。读取时优先看这里有没有存。
如此处理以后，首页的访问时间降到1s一下。

这只是一个临时解决方案，在需求变化很大的时候，用来先实现目标，等稳定以后，重新规划缓存表的结构，并不影响原有架构。
